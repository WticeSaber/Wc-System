**【Epic 史诗故事：2026 世界杯 Elo-泊松比分预测引擎全栈开发】**

**背景描述：** 
基于提供的 sources 与之前的对话记录，该项目旨在构建一个本地化的、零数据成本的足球比分预测引擎及可视化 Dashboard。**底层核心采用 Go 语言结合 Elo 等级分与泊松分布（Poisson Distribution）模型**，**前端则为一个深色科技风格的高保真单页应用（SPA）看板**。以下用户故事（User Stories）已针对 Cursor AI 进行了结构化处理，包含了具体的业务价值、验收标准（AC）与技术上下文，可直接作为 Cursor Composer 的 prompt 逐模块生成代码。

### User Story 1: 零成本数据管道与 ETL 初始化 (后端)
**As a（作为）** 预测引擎后端系统
**I want to（我想要）** 启动时自动从开源渠道拉取并解析历史赛事与 Elo 等级分数据
**So that（以便于）** 系统在**不依赖任何昂贵商业 API 的前提下**，具备最新的球队基本面数据进行数学计算。

**Acceptance Criteria (验收标准)：**
*   **赛事数据拉取：** 利用 Go 标准库 `net/http` 从 GitHub 抓取 `martj42/international_results` 仓库的 `results.csv` 文件。
*   **数据清洗提取：** 使用 `encoding/csv` 流式读取，过滤出各参赛队**最近 10 场有效比赛，独立计算出“场均进球”与“场均失球”**（剥离点球大战等无关维度）。
*   **Elo 数据拉取：** 通过无头 HTTP 请求或 DOM 解析库（如 `goquery`）获取最新的 World Football Elo Ratings，或降级使用 GitHub 开源 Elo 每日镜像（如 `JGravier/soccer-elo` 的 CSV 文件）。

**Cursor 技术上下文提示：**
实现一个内存结构体映射表 `map[string]TeamStats`。注意只关注 90 分钟常规时间的得失球基本面。

### User Story 2: 参数控制台与白盒微操交互 (前端 & 后端)
**As a（作为）** 数据分析师
**I want to（我想要）** 在界面左侧控制台配置比赛参数并手动滑动干预系数
**So that（以便于）** 我可以将战意情报或突发伤病（如核心前锋受伤）通过白盒方式量化融入底层算法中。

**Acceptance Criteria (验收标准)：**
*   **UI 布局：** 采用左右分栏（比例3:7），左侧为暗黑科技风格的参数输入表单。
*   **基础参数：** 提供主客队带搜索的下拉选择框（附带国旗图标），以及一个默认值为 2.5 的“赛事场均进球数”输入框。
*   **高级干预（核心）：** 提供主客队 Elo 手动输入框（留空则默认拉取后端数据），以及**两个严格限制在 `-0.5` 到 `+0.5` 范围内的滑动条（Range Slider），用于主客观微操干预**。
*   **后端对应：** 后端采用 `spf13/cobra` 框架建立 CLI 标志变量（Flags），对应接收 `--avg-goals`, `--home`, `--away`, `--home-mod`, `--away-mod` 等参数。

**Cursor 技术上下文提示：**
前端滑动条的变更需映射到后端的 `beta_A` 与 `beta_B` 系数，如果越界 `-0.5~+0.5` 需抛出错误阻断。

### User Story 3: 核心期望进球数（$\lambda$）数学降维合成 (后端)
**As a（作为）** 算法引擎
**I want to（我想要）** 将 Elo 积分差距、近期攻防状态和主观干预系数合成为独立的期望进球数参数 $\lambda$
**So that（以便于）** 可以将复杂面板数据转化为能够直接输入给泊松分布函数的标准参数。

**Acceptance Criteria (验收标准)：**
*   **Elo 转换：** 提取两队 Elo 积分差，乘以**线性回归斜率系数（$0.0009$ 到 $0.0014$ 之间），计算出“Elo 净胜球修正项”**。**需抹平非东道主比赛的主场优势偏移量**。
*   **攻防乘数：** 主队攻击力系数 = 主队场均进球 / 赛事基准；客队防守弱度系数 = 客队场均失球 / 赛事基准。基础 $\lambda$ = 基准 × 攻击力 × 防守弱度。
*   **终极校准：** 在基础 $\lambda$ 上叠加 Elo 净胜球修正项，再乘以微操干预算式 $(1 + \beta)$。
*   **边界保护：** **代码必须加入软性断言：若经过衰减后的 $\lambda_{Final} < 0.01$，强制平滑至 0.01，防止泊松引擎内存寻址错误或无限大溢出**。

### User Story 4: 泊松矩阵生成与宏观看板可视化 (后端 & 前端)
**As a（作为）** 数据分析师
**I want to（我想要）** 在右侧大盘看板上直观看到 6x6 的概率热力图、胜平负占比及最可能的三种比分
**So that（以便于）** 我能极大降低认知负荷，直接获取决策级数据。

**Acceptance Criteria (验收标准)：**
*   **引擎计算：** 引入 Go 的 `gonum.org/v1/gonum/stat/distuv` 库，分别传入主客队最终 $\lambda$ 实例化 Poisson 结构体。
*   **矩阵推导：** 双重循环计算从 0 球到 5 球的 $6 \times 6$ 交叉联合概率矩阵 $P(i, j)$，**超过 5 球的尾部残差直接忽略**。
*   **宏观胜负：** 使用**“对角线分离法”**，$i>j$ 汇算为主胜，$i<j$ 汇算为客胜，$i=j$ 汇算为平局，并在前端以半环形图或三段进度条展示。
*   **比分可视化：** 前端主体渲染一个 **6x6 热力图（Heatmap）**，颜色深浅与联合概率映射。
*   **Top 3 提取：** 后端调用 Go 原生 `sort` 库将 36 种比分概率倒序排列，前端使用三个领奖台样式的卡片展示“第一顺位”及“候补方案”。

**Cursor 技术上下文提示：**
Gonum 的 `distuv.Poisson.Prob(x)` 是计算独立事件概率的核心。热力图渲染时，需将二维切片数据转换为前端图表库（如 ECharts/Chart.js 或原生 CSS Grid）可读的 JSON 结构。

### User Story 5: 全局极值预警机制 (全栈业务逻辑)
**As a（作为）** 风险控制人员
**I want to（我想要）** 系统在 0-0 平局发生概率超过 15% 时触发全局红色警报
**So that（以便于）** 我能对冲朴素独立泊松模型低估低比分平局的统计学偏差，警惕比赛陷入“窒息式”僵局态势。

**Acceptance Criteria (验收标准)：**
*   **触发条件：** 实时监听推导出的 $P(0,0)$ 概率，一旦**突破 15% 阈值即触发预警**。
*   **前端展示：** 在页面最顶部渲染一条醒目的红色全局警告横幅（Alert Banner），高亮文案提示双方进攻期望极度看衰。
*   **终端展示：** 在 Go 的 CLI 输出中，利用终端颜色库（如 `fatih/color` 或 `pterm`）将相关输出文案置红。
