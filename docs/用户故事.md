**【Epic 史诗故事：2026 世界杯 Elo-泊松比分预测引擎全栈开发】**

**背景描述：** 
基于提供的 sources 与之前的对话记录，该项目旨在构建一个本地化的、零数据成本的足球比分预测引擎及可视化 Dashboard。**底层核心采用 Go 语言结合 Elo 等级分与泊松分布（Poisson Distribution）模型**，**前端则为一个深色科技风格的高保真单页应用（SPA）看板**。

**实现状态：** 全部 5 个 User Story 已于 2026-06-23 完整实现。

---

### User Story 1: 零成本数据管道与 ETL 初始化 (后端)

**As a（作为）** 预测引擎后端系统  
**I want to（我想要）** 启动时自动从开源渠道拉取并解析历史赛事与 Elo 等级分数据  
**So that（以便于）** 系统在**不依赖任何昂贵商业 API 的前提下**，具备最新的球队基本面数据进行数学计算。

**Acceptance Criteria (验收标准)：**
- **赛事数据拉取：** ✅ `internal/adapter/csv_results.go` 使用 `net/http` 从 GitHub 抓取 `martj42/international_results` 的 `results.csv`。支持 `DATA_SOURCE=mock` 切换至本地文件。
- **数据清洗提取：** ✅ `encoding/csv` 流式读取，每队取最近 10 场比赛，独立计算"场均进球"与"场均失球"，剔除点球大战场次。
- **Elo 数据拉取：** ✅ `internal/adapter/elo_ratings.go` 主力使用 `JGravier/soccer-elo` GitHub CSV 镜像（稳定、零反爬限制）。eloratings.net 作为可选的手动补充。
- **附加适配器：** ✅ World Bank GDP API（`worldbank.go`）、Wikimedia 球队介绍（`wikimedia.go`）、FM CSV 球员属性聚合（`fm_csv.go`）均已实现，且失败时优雅降级不阻断主流程。

**技术实现：**
- `internal/cache/store.go`：`DataStore` 在启动时同步初始化，后台 goroutine 按 24h TTL 自动刷新。
- 名称标准化：`internal/adapter/normalization.go` 维护 `canonicalNames` 映射，处理 `"United States"→"USA"`、`"Korea Republic"→"South Korea"` 等变体。

### User Story 3: 核心期望进球数（λ）数学降维合成 (后端)

**As a（作为）** 算法引擎  
**I want to（我想要）** 将 Elo 积分差距、近期攻防状态和主观干预系数合成为独立的期望进球数参数 λ  
**So that（以便于）** 可以将复杂面板数据转化为能够直接输入给泊松分布函数的标准参数。

**Acceptance Criteria (验收标准)：**
- **Elo 转换：** ✅ `internal/engine/elo.go`，`k=0.0011`（可通过 `ELO_SLOPE` 环境变量覆盖，范围 0.0009-0.0014）；中立场：`delta = k × ΔElo`；非中立场：主队 Elo 加 75 分。
- **攻防乘数：** ✅ `internal/engine/stats.go`，`homeAtk = home.AvgGoalsScored / avgGoals`；`awayDef = away.AvgGoalsConceded / avgGoals`；`λBase = avgGoals × homeAtk × awayDef`。
- **终极校准：** ✅ `internal/engine/calculator.go`，叠加 Elo 修正 → 微操干预 `(1+β)` → 可选 DeepSeek 修正 → 软性断言。
- **边界保护：** ✅ `λ_final < 0.01` 时强制平滑至 0.01（`clampLambda()`）。
- **FM CSV 融合：** ✅ `internal/adapter/fm_csv.go` 读取 Football Manager 球员属性，聚合为队级攻防均值，存储于 `TeamRawStats.FMAttributes`（供未来权重融合扩展使用）。

### User Story 4: 泊松矩阵生成与宏观看板可视化 (后端 & 前端)

**As a（作为）** 数据分析师  
**I want to（我想要）** 在右侧大盘看板上直观看到 6×6 的概率热力图、胜平负占比及最可能的三种比分  
**So that（以便于）** 我能极大降低认知负荷，直接获取决策级数据。

**Acceptance Criteria (验收标准)：**
- **引擎计算：** ✅ `internal/engine/poisson.go`，使用 `gonum.org/v1/gonum/stat/distuv.Poisson`，分别传入主客队最终 λ。
- **矩阵推导：** ✅ `BuildMatrix()` 双重循环生成 6×6 联合概率矩阵；超过 5 球的尾部直接忽略。
- **宏观胜负：** ✅ `ExtractOutcomeProbs()` 对角线分离法，结果归一化后在前端以 SVG 半环形图 + 三段进度条展示。
- **比分可视化：** ✅ 前端 `Dashboard.tsx` 渲染 6×6 热力图，颜色深浅与联合概率线性映射。
- **Top 3 提取：** ✅ `TopNPredictions()` 使用 `sort.Slice` 倒序，前端领奖台样式卡片展示。
- **API 接口：** ✅ `POST /api/predict` 返回完整 `PredictionResult` JSON（详见 `docs/API.md`）。

### User Story 5: 全局极值预警机制 (全栈业务逻辑)

**As a（作为）** 风险控制人员  
**I want to（我想要）** 系统在 0-0 平局发生概率超过 15% 时触发全局红色警报  
**So that（以便于）** 我能对冲朴素独立泊松模型低估低比分平局的统计学偏差，警惕比赛陷入"窒息式"僵局态势。

**Acceptance Criteria (验收标准)：**
- **触发条件：** ✅ `internal/engine/poisson.go`，`CheckAlertThreshold()` 检查 `matrix[0][0] >= 0.15`。
- **前端展示：** ✅ `Dashboard.tsx` 顶部渲染红色全局警告横幅（`animate-pulse` 动画）。
- **终端展示：** ✅ `cmd/calc.go` `printCalcResult()`，使用 `fatih/color` 将预警文案置红。
- **阈值说明：** 与 PRD 保持一致（15%），对应 `poisson.go` 中 `alertThreshold = 0.15` 常量。

### User Story 6（新增）: DeepSeek 语义情报层 (后端)

**As a（作为）** 高级数据分析师  
**I want to（我想要）** 通过 AI 分析双队近期比赛数据，自动生成一个情报修正系数  
**So that（以便于）** 将 LLM 的语义理解能力融入量化预测，对冲历史统计数据的时效性偏差。

**Acceptance Criteria (验收标准)：**
- **AI 调用：** ✅ `internal/semantic/deepseek.go`，通过 OpenAI-compatible API 调用 `deepseek-chat` 模型，`temperature=0.2`。
- **返回格式：** ✅ Prompt 约束输出为 `{"modifier": float, "reasoning": "..."}` 纯 JSON，边界保护 clamp 至 `[-0.2, +0.2]`。
- **集成位置：** ✅ `calculator.go` Stage 3，在 `(1+β)` 微操修正之后、`clampLambda` 之前应用。
- **优雅降级：** ✅ API 失败时返回 `modifier=0.0` 并 warn 日志，不中断计算流程。
- **前端展示：** ✅ `Dashboard.tsx` 在数据状态栏显示 `AI MODIFIER: +0.0500` badge。
